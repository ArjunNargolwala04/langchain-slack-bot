# Overview

I built a slack-based Q&A chatbot that answers questions regarding a startup called Northstar’s internal database, including information about customer records, call transcripts, support tickets, and internal communications. This was built on top of LangGraph in python, via FastAPI, and using GPT-4o as a base model. The agent utilizes a ReAct loop, and is capable of handling single-customer lookups and complex cross-customer analysis. It answers all 7 example queries correctly, with an avg of 2.3 tool calls per query at roughly $0.05 on each query.

# Architecture 

I broke the project down into two main layers. The first of which is the agent layer, which holds all of our intelligence related code. It allows us to do LLM reasoning, tool execution, retrieval strategy, and conversation memory, among a few extra things. The second layer is the app layer, which allows us to deploy within the Slack interface, and holds the code for things like webhook handling, auth, formatting, and async processing. These layers connect through agent.stream(). Separation was due to testing and modularization purposes. We also can then run our eval harness, the local test scripts, and the full test suite against our agent directly. 

Our graph has 4 nodes. Agent node calls the LLM, which then proceeds to either request some tool calls, or produces a final answer ready to be delivered to the end user. Tool nodes are built to execute whatever db operation the LLM has requested, and then returns the result. Our verify node runs only after a final answer is produced, and gives a confidence score based on the tool trace and outputs. And finally, we have a limit node which triggers if our agent exceeds 15 tool calls, injecting an exit message rather than looping endlessly. 

Our multi-turn conversation function works by mapping Slack's thread_ts to LangGraph's thread_id. Having the same thread in Slack means having the same conversation state in the agent, and we use an SqliteSaver checkpointer to persist this state across restarts.

# Key Decisions

LangGraph over DeepAgent: We were given two options for this project. DeepAgents is generally designed for long-running, autonomous agents with deeper more complex tasks, and usually have file system access and sub-agents doing a hierarchically assorted task list. For this task, we actually prefer to have tight control over our retrieval loop. We needed to search db, read very specific documents, and then string together an answer. LangGraph gives us the correct primitives to do this better. We can add custom nodes like verify and limit, hook into streaming, and map checkpoints to Slack threads. DeepAgent is a much better fit for multi-stage long-form tasks, things like file generation, or coordination between specialized agents. Our task required precision over a single retrieval path, which is why LangGraph excels for our use case.

Manual Graph over Create React Agent: There is an inbuilt create_react_agent in LangGraph, but I chose to build it manually instead. This offers more control, allowing me to create verify and limit nodes, as well as get progressive Slack status updates via agent.stream(). Also it was far easier to debug and explain things, as every node in the graph exists for a distinct reason. 

Two-phase Retrieval: The db contains 250 text-rich artifacts, loading full content each time into context would be horrendously inefficient, and degrade quality. Instead, I chose to do 2 phase retrieval. First running search_artifacts to get titles, types, and summaries. Agent then decides which are relevant and useful. Then read_artifact returns full content for the chosen artifacts, capped at 5 per call. This is our core efficiency mechanism on tool calling. Easy queries usually use 2 tool calls (search, and read), and hard ones usually use between 3-5. 

FTS5 over embeddings/RAG: The db ships with an FTS5 full-text search index. My initial instinct was to use RAG with vector embeddings, but for this specific case, FTS is much much faster and is deterministic. No extra infra needed, no embedding model, no vector store, no chunking strategy, all of it taken care of (in a different case I would have done exactly this sort of RAG pipeline though). Our tradeoff here is that FTS requires keyword matching, which can break on differing customer inputs. The fix for that is the OR fallback. When an AND query returns zero results, our tool automatically retries with OR between terms so that any matching keyword surfaces results. This one change is what made the hard cross-customer queries work consistently. The date auto-quoting in the search tool addresses a related issue to this. FTS5 does treat hyphens as operators, so search for 2026-02-20 gets parsed as three separate tokens. The tool detects date patterns with a regex and wraps them in quotes to solve this.

Prompt Engineering: Honestly, the largest bump in accuracy came from more and more refined system prompts. I chose to encode specific retrieval strategies for differing query types, and it yielded great results. For single-customer lookups using SQL LIKE for partial name matching. For cross-customer or pattern detection, exhaustively query all relevant customers by attribute before concluding. For competitor defection, compare multiple reports to find the most explicit threat rather than stopping at the first hit. The tools are just generic database operations, but this engineered prompt is what ended up making them really effective for this data/question set.

Confidence Scoring: After our agent produces an answer, we use a verify node to make a second LLM call to score confidence w/ justification. I wanted to avoid self-assessment as it is generally unreliable in my experience, and a separate evaluation step w/ focused verification prompt produces more calibrated scores. The tradeoff here is latency, adding about 1-2 seconds per query. However in the context in which we are producing answers, the cost of being wrong and then being wrong in a chain of messages far outweighs the mild delay in final answer production greatly. And it functions as a real trust signal for users, which for me is great UX. 

# Slack Integration and Security

Slack requires a 200 response within 3 seconds of a webhook event, and usually our agent takes anywhere from 5-15 seconds to produce a final answer. So I solved this using an async pattern. Acknowledging the webhook immediately and displaying a summary action message while running the agent in the background improves both the UX, and complies with this Slack API response time parameter. This runs via asyncio.create_task, and the placeholder updates as the action running changes. And then the final LLM response is post-processed in code and stripped of markdown format to remove the ugly formatting it usually displays within Slack. 

Every single incoming webhook request is verified using HMAC-SHA256 signature validation against the app signing secret. We use hmac.compare_digest for constant time comparison, to prevent any timing attacks, and reject any requests with timestamps older than 5 mins. As well as this, the db tools enforce strict read-only access by rejecting any SQL query that doesn’t begin with SELECT. Secrets are loaded from env vars, never ever hardcoded. Error handling also does distinguish between rate limits from OpenAI, auth, failures, db errors, and generic exceptions, returning a user-facing message for each of these cases. 

# Observability and Evaluation

I really wanted to integrate some eval and observability features into my agent, especially with LangSmith, which is really easy to integrate into the project. Also serves as a really good baseline for traces and reward signals, should we ever want to make this an RL agent etc. 

As such, every tool call is logged with some structured data. We store tool name, args (truncated), execution time, and result size in chars. This runs in all environments and appears in server logs, giving us a traceable record of the agent's reasoning path for any query. And when the LangSmith API key is set in the env, LangSmith tracing also auto-activates. Every graph call is traced w/ full node-level detail. Also required zero extra code, LangChain integration works great behind the scenes. We then get a full trace of the agent's complete reasoning path. 
The eval harness runs all 7 example queries and produces a structured performance report. Giving correctness (+/- keyword match against expected), tool call count, wall-clock time, tokens, and estimated cost based on current API fees. 

I built this eval infra as the first part of an agent improvement loop. It defines a reward signal, namely correctness and efficiency. LangSmith captures trajectories. In prod, we could also close the loop by collecting user feedback as an additional signal. Slack reactions and explicit corrections would combine with the automated evals to feed prompt optimization or fine-tuning. I used similar frameworks in RLVR for coding agents, previously, and so I wanted to integrate it here. 

# Potential Improvements

Query Classification Node: A lightweight step before ReAct loop to identify query type and route each through a tailored prompt. This is rather than packing all retrieval strategies into one system prompt and hoping the LLM picks the right one. Not needed to achieve 100% accuracy for this 250 artifact db list, but if we scaled up complexity then definitely necessary. 

Conversation Summaries: Right now, full message history is passed on every subsequent turn. If 10+ turns, getting a periodic summary would be of great use. 
Artifact Caching: For follow-up questions about the same customer, I would just re-fetch artifacts the agent already retrieved. A within-thread cache in the graph state would eliminate some redundant tool calls on follow-ups, again, needed for larger scale/complexity dbs.

# Takeaways

I tried to do a couple of things with this project. Play to my strengths in terms of getting a clean and working agent loop, and then laying the groundwork for RL/evals/observability. I think within a more complex agent use case, that would have been really useful and ultimately necessary. For this simpler case though, I really focused on end user ease, and overall accuracy/efficiency. I think in the product context of a Slack bot, you really just want a feature set that is going to be advantageous in terms of trust, speed, and aesthetics, and I tried to encode that in this project.
